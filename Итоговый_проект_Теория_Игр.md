## Пример проекта: CallCenterOpt

> Пример аналогичного проекта приведён в файле `Proekt_TG_26.pdf`

**Название:** CallCenterOpt – Симулятор колл-центра с оптимизацией штата

**Цель:** Разработать Python-приложение, моделирующее работу колл-центра как системы массового обслуживания, и подобрать оптимальное количество операторов, минимизируя затраты на персонал при соблюдении SLA (вероятность ожидания в очереди ≤ 15%). Выбор стратегии найма формализуется как игра в условиях риска (неопределённый поток звонков).

### Математические компоненты

| Компонент | Описание |
|-----------|----------|
| **СМО (M/M/c)** | Входящий поток звонков: пуассоновский с интенсивностью λ. Время обслуживания: экспоненциальное со средним 1/μ. Количество каналов: c (операторы). Расчёт: P_wait (вероятность ожидания), L_q (длина очереди), W (среднее время ожидания). |
| **Игра с природой (риск)** | «Природа» выбирает сценарий нагрузки: {низкая λ=20, средняя λ=50, высокая λ=90}. Колл-центр выбирает стратегию: {c=3, c=5, c=7}. Платёжная матрица строится на основе штрафов за ожидание + затрат на операторов. Решение: критерий Гурвица или Байеса (задаётся пользователем). |
| **Линейная оптимизация** *(опционально для сравнения)* | Минимизация `Cost = c * salary + α * P_wait` при ограничении `P_wait ≤ 0.15`. Решается перебором или простым ЛП-солвером для валидации. |

### Архитектура ПО

```
callcenter_opt/
├── config/          # scenario.yaml, params.json
├── core/
│   ├── queue_math.py    # Формулы Эрланга C, расчёт метрик СМО
│   ├── game_solver.py   # Построение матрицы, критерии Гурвица/Байеса
│   └── models.py        # Pydantic-схемы для валидации параметров
├── api/             # FastAPI роуты (опционально) или CLI (click)
├── tests/           # pytest: unit для формул, integration для сценариев
├── ui/              # Streamlit или Rich-консольный интерфейс
├── requirements.txt
└── README.md
```

**Паттерны:** Parameter Object (для сценариев), Strategy (критерии принятия решений), Repository (чтение конфига)

**Сетевое взаимодействие:** CLI по умолчанию + опциональный FastAPI-сервер для веб-доступа

**Контейнеризация:** Один Dockerfile (не обязателен, но приветствуется)

### Реализация и технологии

1. NumPy, SciPy.stats (распределения, формулы Эрланга)
2. PuLP или ручной перебор для ЛП-валидации
3. Pydantic (валидация входных параметров)
4. Click или Streamlit (интерфейс)
5. pytest, pytest-cov (тестирование)
6. black, ruff (стиль кода)

### Этапы разработки

- **Сбор требований:** Определение метрик SLA, сценариев нагрузки, формулы затрат
- **Математическое ядро:** Реализация `queue_math.py` с проверкой на известных примерах (например, λ=10, μ=2, c=3)
- **Игровой модуль:** Генерация платёжной матрицы, реализация критериев Гурвица/Байеса/Вальда
- **Интерфейс:** CLI/Streamlit для ввода параметров, выбора критерия, вывода таблицы решений
- **Тестирование:**
  - Unit: проверка формул СМО против аналитических значений
  - Integration: запуск полного сценария, проверка формата вывода
  - Покрытие: ≥65%

**Демонстрация:** Запуск 3 сценариев, сравнение стратегии по Гурвицу и оптимальной по ЛП

### Ожидаемые результаты

- Корректный расчёт характеристик M/M/c (проверено на эталонных таблицах)
- Выбор стратегии найма с обоснованием (например, c=5 по критерию Гурвица при α=0.6)
- Снижение вероятности ожидания до ≤15% при оптимальном c
- Воспроизводимый запуск одной командой, понятная структура кода, тесты, документация
